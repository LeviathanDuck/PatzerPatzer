## Context

Bug report:
- on the Opening Tree page, when the window width reaches certain sizes, the sidebar /
  openings list appears to disappear
- the user wants to know whether this behavior is explicitly present in the code and how
  to stop it from happening

The important question is not just "make it look better."
You must determine whether the disappearance is:
- an intentional responsive-collapse rule already in code
- an unintended grid/layout squeeze
- or a broader shell/layout ownership issue

## Relevant current code areas

Start with the actual current ownership points:
- `src/openings/view.ts`
- `src/styles/main.scss`
- `src/openings/ctrl.ts`

Known current suspects from repo inspection:
- the openings session shell is rendered in `renderOpeningsSession()` and uses
  `.openings__session-body`
- the left tool rail is rendered by `renderToolRail()`
- the openings session layout in `src/styles/main.scss` uses a three-column grid for:
  - tool rail
  - board
  - session panel
- there is also a responsive breakpoint:
  - `@media (max-width: 700px) { .openings__session-body { grid-template-columns: 1fr; } }`

## Required workflow

1. Inspect the actual current Patzer code first.
2. Identify exactly which user-facing surface is disappearing:
   - left tool rail
   - right-side Opening Tree move list / session panel
   - or both
3. Determine whether the behavior is explicitly encoded in CSS/layout rules or is an indirect
   consequence of the current grid math.
4. Inspect relevant Lichess source if it helps with board-plus-side-panel layout decisions.
   If Lichess has no direct equivalent for this openings shell, say that clearly instead of

7. If the cause is ambiguous or needs broader design decisions, stop after diagnosis and give a
   precise recommendation instead of making a risky layout rewrite.
