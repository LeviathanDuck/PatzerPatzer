# CCP-649 — Polish next phase prompt creation panel

Read and follow:
- `/Users/leftcoast/Development/PatzerPatzer/CLAUDE.md`
- `/Users/leftcoast/Development/PatzerPatzer/AGENTS.md`

## Context

Codex has already implemented the backend contract for the `Generate Next Prompts For Sprint`
panel in sprint detail.

The panel is now conditionally available only when:
- the sprint is normalized enough for reliable next-phase generation
- a next available phase exists
- that phase still contains at least one task with `No Prompt Exists Yet`

The backend also already provides:
- `nextAvailablePhase`
- `tasksNeedingPrompts`
- `nextPhaseCoveredTasks`
- `nextPromptsTemplate`

The current presentation is functional, but it still reads like raw tooling instead of a polished
operator surface.

## Relevant file

- `scripts/generate-prompt-dashboard.mjs`

Generated output to inspect:
- `docs/prompts/dashboard.html`

## Goal

Improve the design and information hierarchy of the `Generate Next Prompts For Sprint` panel so
it is obvious, high-signal, and easy to trust.

Focus on:
- making the panel feel like the main action surface when it appears
- clarifying that it is scoped to the next available phase
- helping covered vs uncovered tasks read clearly
- making the copy action feel primary

## Hard constraints

- Do not change the backend rules for when this panel appears
- Do not change the generated template semantics
- Do not invent new sprint state logic
- Keep the changes scoped to this action panel and its immediate presentation

## Backend contract already implemented

Consume the existing backend fields as-is:
- `showNextPromptPanel`
- `nextAvailablePhase`
- `tasksNeedingPrompts`
- `nextPhaseCoveredTasks`
- `nextPromptsTemplate`

Do not rewrite backend derivation logic in this prompt.

## Validation

Run:
```sh
npm run dashboard:generate
```

Open `docs/prompts/dashboard.html`, open sprint details where the panel appears, and verify:
- the panel clearly reads as the next action for the sprint
- the next available phase context is visually clear
- tasks needing prompts vs already-covered tasks are easier to distinguish
- the copy action is primary and obvious
