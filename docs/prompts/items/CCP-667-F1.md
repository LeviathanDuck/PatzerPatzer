# CCP-667-F1 — Replace Separate Diff Preview With Line-By-Line Stylized Editor

## Summary

Follow up on `CCP-667` by replacing the current textarea-plus-separate-preview editing model
with a single line-by-line stylized diff editor.

Scope split to avoid overlap:
- `CCP-667` remains responsible for making prompt metadata lines truly locked/non-editable
- `CCP-667-F1` is responsible for the new live editor model itself across both prompt detail
  editing and sprint detail panel editing

## Problem

The current implementation does not match the intended editing UX:
- edit mode still uses a normal textarea
- diff is shown in a separate preview panel
- the main editor does not itself become the visual diff surface

The target behavior is simpler than full track-changes editing:
- line-by-line diff, not character-by-character editing logic
- one unified stylized editor surface
- no separate live preview panel

## Required Behavior

Apply this to both:
- prompt detail editing in the `Prompts` tab
- sprint panel editing in sprint detail view

### Editing Model

- edit mode uses one stylized editor block
- unchanged lines remain visible
- no separate `Live Diff Preview` panel remains
- line numbers are visible

### Line Activation

When the user activates editing on an unchanged line:
- that line immediately becomes:
  - the original line shown in red struck-through form
  - a green editable replacement line directly below it
- the new green line is prefilled with the original text
- the user edits only the green replacement line from there

### Added / Deleted / Replaced Lines

- inserted lines appear as green editable lines in place
- deleted lines remain visible in red struck-through form
- full-line deletions remain visible until save/reset/restore
- each deleted line gets its own restore control
- restoring a deleted line returns it to a normal unchanged line
- if replacement lines exist for that deleted line, keep them unless they are blank

### Locked Lines

For prompts:
- registry-owned metadata stays inside the same editor block at the top
- those lines are locked and non-editable

For sprint panels:
- the sprint context header stays at the top of the same editor block
- it remains visible but is not part of the diffable editable body

Locked-line cue:
- each locked line shows a small lock icon on the far left

### Read-Only Saved View

After save:
- the main read-only view shows the stylized diff against the immediately previous saved version
- unchanged lines remain visible in that view
- archived history still exists separately and remains oldest-first

### Copy Behavior

Copy from either edit mode or read-only diff view must copy:
- the clean current body only
- never the red deleted lines or visual diff markup

### Save Confirmation

Replace the current text-only overwrite confirmation with an inline visual save-confirm state:
- the editor surface becomes temporarily non-editable during confirmation
- available actions:
  - `Confirm Save`
  - `Back to Editing`
  - `Copy`
- if save fails:
  - return to editable mode automatically
  - preserve the attempted changes
  - show an error message

### Resets / Cancel

- `Reset to Original` or `Reset to Saved` clears all diff markers immediately
- `Cancel` discards edits and returns to read-only saved view

## Constraints

- do not implement character-by-character tracked editing
- keep the implementation line-based
- do not regress localhost-only behavior for write actions
- do not break superseded prompt archive handling
- do not break sprint panel saved-body history handling
- do not let locked prompt metadata become editable

## Validation

- open an editable prompt and confirm edit mode uses one stylized line-diff editor with no separate live preview panel
- activate editing on one unchanged prompt line and confirm it becomes one red original line plus one green editable replacement line
- confirm prompt metadata lines stay visible, locked, and marked with lock icons
- save a prompt edit and confirm read-only view shows diff against the immediately previous saved version
- open a sprint panel editor and confirm it uses the same line-based stylized editing model while keeping the sprint header non-diffable and locked
- copy from edit mode and read-only mode and confirm clipboard text is the clean current saved body only
- confirm static `file://` mode still instructs the user to run `node server.mjs` and use `http://localhost:3001/dashboard`
