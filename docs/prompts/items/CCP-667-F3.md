# CCP-667-F3 — Fix Sprint Panel Editor Flicker Before Entering Edit View

## Summary

Follow up on `CCP-667-F1`.

The same edit-transition glitch fixed in `CCP-667-F2` for prompt detail also appears in
the sprint detail editable panels.

Right now, when the user clicks `Edit` on a sprint action panel, the UI briefly shows an
intermediate preview/state with another edit button before it settles into what looks like the
real editor.

On further inspection, the problem is worse than a visual flicker:
- the user may see an extra preview-like step before edit mode
- even after pressing through that extra step, the resulting surface may still not behave like a
  real editable text editor
- the user should not have to click through a preview phase just to reach actual editable text

This should feel immediate and intentional:
- clicking `Edit` should go straight into the stylized sprint panel editor
- the sprint detail view should not flash a preview-like intermediate state first

## Required Fix

Implement the smallest safe fix so sprint panel edit mode enters cleanly.

Expected behavior:
- click `Edit` on a sprint panel
- that panel goes directly into a real editable stylized text viewer
- no extra preview frame or duplicate edit step appears first
- once edit mode is entered, the panel body must actually accept text editing immediately

Constraints:
- keep this scoped to sprint panel edit-mode entry behavior
- do not redesign the sprint panel editor again
- do not regress saved-body history, cancel/reset, save confirm, or localhost-only behavior
- do not regress the prompt-side fix from `CCP-667-F2`
- only touch shared helpers if that is the smallest safe way to fix the sprint side
