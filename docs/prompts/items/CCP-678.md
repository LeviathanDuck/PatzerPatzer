# CCP-678 — Add Sprint Recompute To Dashboard Refresh Button

Prompt ID: CCP-678
Task ID: CCP-678
Source Document: user request 2026-04-02

## Context

The dashboard Refresh button currently calls `fetchLiveData()` to reload prompt and sprint
data from the server. However, it does not trigger `sprint:recompute`, which means the sprint
registry state (task execution rollups, phase rollups, sprint rollups) can drift from reality
without the user noticing. When the user clicks Refresh, they expect a fully up-to-date view
of sprint progress — not a stale snapshot.

## Task

Add a `/api/sprint-recompute` POST endpoint to `server.mjs` that runs
`recomputeSprintRegistryState()` and `refreshSprintArtifacts()`. Then update the live-mode
Refresh button handler in `scripts/generate-prompt-dashboard.mjs` to call this endpoint
before `fetchLiveData()` so that every manual refresh also syncs sprint state.

### 1. Add the server endpoint

In `server.mjs`, add a handler for `POST /api/sprint-recompute`:

- Call `recomputeSprintRegistryState()` inside `mutateSprintRegistryLocked` (same pattern as
  other sprint mutation endpoints)
- Call `refreshSprintArtifacts()` after the mutation to regenerate SPRINT_STATUS.md and
  dashboard HTML
- Return `{ ok: true }` on success
- Return a 500 JSON error on failure

### 2. Update the dashboard Refresh button handler

In `scripts/generate-prompt-dashboard.mjs`, find the Refresh button click handler in the
live-mode JS. It currently calls `fetchLiveData()`. Change it to:

1. Show a loading/working state on the button (e.g., button text → "Refreshing…", disabled)
2. POST to `/api/sprint-recompute`
3. On success or failure, call `fetchLiveData()` as before
4. Restore button state after `fetchLiveData()` resolves

The sprint recompute failure must NOT block the page refresh — if the POST fails, log a
console warning but still proceed with `fetchLiveData()`.

### Files to touch

1. `server.mjs` — add `/api/sprint-recompute` endpoint
2. `scripts/generate-prompt-dashboard.mjs` — update Refresh button handler
3. `docs/prompts/dashboard.html` — regenerated output (via `npm run dashboard:generate`)

## Validation

1. Open the dashboard in live mode (`node server.mjs`)
2. Click Refresh — observe button briefly shows "Refreshing…"
3. Confirm the sprint registry state is recomputed (check terminal output for recompute log)
4. Confirm the dashboard reloads with updated sprint data
5. Confirm that if the server is not running (static mode), behavior is unchanged
