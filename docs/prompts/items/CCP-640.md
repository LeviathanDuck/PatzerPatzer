# CCP-640 — Fix eval diff boxes not populating in Learn From Mistakes

## Context

The "vs Engine Best" and "vs Move Played" eval diff boxes added in CCP-632–635 are not
populating on most moves during Learn From Mistakes sessions. The engine IS running and
completing evaluations (confirmed via console), but the displayed boxes show no values.

The bug is almost certainly a timing problem: the snapshot of `evalCache` entries is captured
synchronously at move-play time inside `retroMoveHandler.ts`, but the engine has not yet
evaluated the positions that the snapshot reads from.

## Current flow (broken)

1. User plays a move in retro mode — `retroMoveHandler.ts` fires `afterMove` hook
2. `afterMove` immediately reads three `evalCache` entries:
   - `evalCache.get(ctrl.path)` — solving move position (engine hasn't evaluated this yet)
   - `evalCache.get(cand.parentPath)` — pre-move baseline (may have batch eval, may be stale)
   - `evalCache.get(cand.path)` — game mistake position (may have batch eval, may be stale)
3. Snapshot is written with whatever (often `undefined`) values were found
4. The live engine later evaluates these positions and calls `_onLiveEvalUpdated`, but the
   filter in `main.ts` only watches for `cand.parentPath` updates — it does not update the
   snapshot when the solving move position (`ctrl.path`) completes
5. Result: boxes render as blank or missing on nearly every move

## Likely fix

The correct approach is to make the snapshot **reactive to engine updates**:

Instead of capturing a one-shot snapshot at move-play time and never updating it, the
`setOnLiveEvalUpdated` callback in `main.ts` should be expanded to also handle updates
for ALL three positions involved:

- `cand.parentPath` — engine best baseline
- `ctrl.path` — solving move eval (the position the user navigated to)
- `cand.path` — game mistake move eval

When any of these three paths receives a live engine update, the relevant field(s) in the
snapshot should be refreshed and `redraw()` called.

An alternative approach: re-read `evalCache` lazily in `renderDualEvalBoxes` at render time
(passing evalCache into the view), instead of relying on a one-shot snapshot. This avoids
timing issues entirely.

Choose whichever approach is simpler and more correct. Document the choice.
