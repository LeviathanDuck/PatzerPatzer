# Create shared eval diff utility — `src/analyse/evalDiff.ts`

## Context

Part of CCP-595 (eval diff for retro move quality). This is child prompt 1/3.

"Learn From Your Mistakes" needs to display how much better the engine's best
move was vs. the move actually played. Before wiring this into any view, we
need a clean, reusable utility that computes the diff from eval cache data.

The utility lives in `src/analyse/` (not `src/engine/`) because it is an
analysis-domain concept — it relates positions to each other, not to UCI
protocol. It will be importable by puzzles and other tools later.

## Task

Create `src/analyse/evalDiff.ts`. No other files are touched.

## Interface to implement

```typescript
/**
 * The computed difference between the engine best move and the played move,
 * expressed in both win-chance and centipawn terms.
 *
 * cpDiff and wcDiff are from the mover's perspective:
 *   positive = engine best is better for the mover than what was played.
 *   negative = the played move was actually better (rare; indicates eval noise).
 */
export interface EvalDiff {
  /** Win-chance diff, 0–0.5 scale. Mirrors the existing `loss` field on RetroCandidate. */
  wcDiff: number;
  /**
   * Approximate centipawn diff (full 100cp = 1 pawn scale).
   * Derived from win-chance diff via inverse sigmoid approximation.
   * May be null if the source eval has no cp data (mate-only eval).
   */
  cpDiff: number | null;
  /**
   * Human-readable pawn string, e.g. "+2.0", "+0.5", "mate".
   * Formatted for display; caller decides surrounding copy.
   */
  formatted: string;
  /** Whether the source is high-confidence (derived from cp) or approximate (from wc only). */
  confidence: 'cp' | 'wc-approx';
}

/**
 * Input evals for the diff computation.
 * All values are stored in Patzer's evalCache format.
 */
export interface EvalDiffInput {
  /**
   * Eval at the parent position (the exercise start).
   * cp, if present, is from the side to move at the parent position.
   */
  parentEval: { cp?: number; mate?: number; loss?: number };
  /**
   * Eval at the child position (after the played move).
   * cp, if present, is from the side to move at the child position
   * (i.e., the OPPONENT of the original mover).
   */
  nodeEval: { cp?: number; mate?: number; loss?: number };
  /** Which color played the move being evaluated. */
  playerColor: 'white' | 'black';
}

/**
 * Compute the eval diff between the engine best and the played move.
 *
 * Primary source: nodeEval.loss (precomputed mover-perspective win-chance loss
 * from batch analysis). This is the preferred source because it was computed
 * from both evals in a single analysis pass, avoiding depth-inconsistency.
 *
 * Secondary source: parentEval.cp + nodeEval.cp — used only when loss is absent.
 * In Patzer's eval cache, cp values are stored from the side to move's
 * perspective (Stockfish UCI convention). The child node has the opponent
 * to move, so its cp must be negated to convert to the original mover's
 * perspective.
 *
 * Returns null if neither source yields a usable value.
 */
export function computeEvalDiff(input: EvalDiffInput): EvalDiff | null

/**
 * Format a pawn value for display.
 * Examples: 0.5 → "+0.5", 2.0 → "+2.0", 0.0 → "0.0"
 * Negative values (played move better than engine best) formatted as-is.
 */
export function formatPawns(pawns: number): string
```

## Implementation notes

### Win-chance to approximate centipawn conversion

The `loss` field is on the 0–0.5 win-chance scale (halved). To get an approximate
centipawn equivalent, invert the logistic function used by `evalWinChances`:

```
wcDiff (on full 0–1 scale) = nodeEval.loss * 2
cpApprox = ln(wcFull / (1 - wcFull)) / 0.005   (inverse of the standard Stockfish sigmoid)
```

Cap the output at ±1500 cp (= ±15 pawns) to avoid astronomical numbers from
near-decisive evaluations. For positions where `parentEval.mate` or
`nodeEval.mate` is set, return `formatted: 'mate'` and `cpDiff: null`.

### cp-based path (secondary)

If `nodeEval.loss` is undefined but both cp values exist:

```
// Convert both to original mover's perspective:
moverParentCp = playerColor === 'white' ? parentEval.cp : -parentEval.cp
moverNodeCp   = playerColor === 'white' ? -nodeEval.cp  : nodeEval.cp
cpDiff = moverParentCp - moverNodeCp
```

Note: this path is less reliable because parent and child evals may have been
computed at different depths. Prefer `loss`-based when available.

### Negative diffs

A negative diff means the played move was evaluated as better than `parentEval`
predicted. This can happen due to depth noise. Clamp to 0.0 for display
(show "0.0" rather than "-0.1") but preserve the raw value in the struct.
