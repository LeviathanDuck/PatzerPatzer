# CCP-654 — Investigate: skip opponent import dialog and show loading bar inline

## Context

Currently, importing an opponent in the Opponents tool shows a dialogue (modal or
panel) before or during the fetch. Separately, once an opponent's games load, the
opening tree build triggers a visible loading bar in a fixed location in the view.

The question is: **should the import dialogue be removed entirely, and should the
initial opponent-fetch loading bar be rendered in the same place the opening tree
loading bar already lives?**

This is a research and investigation prompt. No code changes are made. The output is
a written recommendation — yes, no, or conditional — with evidence from the current
codebase and a clear rationale.

## Investigation scope

### 1. Audit the import dialogue

Find where the opponent import dialogue/panel/modal is rendered and triggered.

Answer:
- What file(s) render it?
- Is it a modal overlay, an inline panel, or something else?
- What state triggers it to open / close?
- What does it do that an inline flow would need to replicate? (e.g. username input,
  platform selection, error display, cancel affordance)
- Is there a case where the user might need to interact with it before the fetch can
  start (e.g. entering a username they haven't yet set)?

### 2. Audit the opening tree loading bar

Find where the opening tree loading bar is rendered.

Answer:
- What file renders it and what class / vnode structure?
- What state variable drives it (e.g. a boolean flag, a progress value)?
- Where in the layout does it appear — is it a fixed position within the Opponents
  view, or somewhere more global?
- How wide is its display lifecycle? (e.g. visible only during tree build, or also
  during game fetch?)

### 3. Assess the proposed change

Given the findings above, evaluate:

- **Feasibility**: can the import dialogue's function be absorbed into the inline
  loading bar area without losing any user-facing capability?
- **State coverage**: does the loading bar location have enough room to show fetch
  progress, errors, and a cancel affordance? Or does it only show a spinner/bar?
- **UX quality**: does removing the dialogue make the flow feel faster and more
  direct, or does it remove useful friction (e.g. confirming the username before
  a long fetch)?
- **Edge cases**: what happens if the username is not yet set and the user navigates
  to the Opponents tool? Does the dialogue serve as the username-entry point in that
  path?
- **Risk**: are there any states where removing the dialogue would leave the user
  with no feedback or no way to cancel a long-running fetch?

### 4. Lichess reference check

Look at `ui/learn/src/` or any analogous flow in the Lichess source where a fetch or
import is initiated without a modal — is there a pattern for inline progress
replacing a pre-fetch dialog?

## Output

Write a structured recommendation with the following sections:

### Findings
Bullet-point summary of what each audit found (dialogue, loading bar, Lichess ref).

### Recommendation
One of:
- **Yes — implement the change**: rationale + any preconditions
- **No — keep the dialogue**: rationale + what would need to change before this
  could be reconsidered
- **Conditional**: what conditions must be met first

### If yes: rough implementation notes
- Which files to touch
- What state changes are needed
- Whether existing loading bar state can be reused or extended
- Any new affordances needed (error display, cancel) that aren't currently there

Write the recommendation as a comment in the prompt body below the `## Output`
header, or in a short `docs/reference/` note if it is substantial.

**Do not make any code changes.** Research only.
