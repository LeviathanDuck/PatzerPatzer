## Context

Part 1 of 4 in the retro live eval diff feature (manager: CCP-636).

The retro session currently stores a static `bestMove` from batch analysis on each
`RetroCandidate`. This prompt extends `RetroCtrl` to:

1. Track the live engine's current best move and eval for the active candidate position,
   updating as engine depth increases.
2. Expose the live best move and its eval so downstream code can use whichever is deeper
   (live vs batch).
3. Fire a callback the moment the live engine determines the original game move was
   actually the best move available — the "vindication" event.

## Target files

- `src/analyse/retroCtrl.ts`
- `src/analyse/retro.ts` (add `gameMoveSan` field to `RetroCandidate` if not present)

## Changes required

### src/analyse/retro.ts

Add `gameMoveSan: string` to `RetroCandidate` if it is not already there. This is the
SAN of the move played in the game (the mistake). It is needed by the view to display
the label "Move played in game" without requiring a UCI→SAN conversion at render time.

Check first — `playedMoveSan` already exists. If it covers this use case, no change needed.

### src/analyse/retroCtrl.ts

Add the following to the `RetroCtrl` interface and implementation:

```typescript
// Live engine best move for the active candidate position.
// Updated whenever the engine reports a new best move at higher depth.
// Starts as the batch best move from the candidate.
liveBestMove(): string | undefined;   // UCI
liveBestEval(): { cp?: number; mate?: number; depth: number } | undefined;

// The eval of the move the user just played during solving (UCI).
// Set by setSolvingMoveEval(); cleared when advancing to a new candidate.
solvingMoveEval(): { cp?: number; mate?: number } | undefined;

// Called by main.ts whenever the engine reports a new line for the active position.
// Updates liveBestMove/liveBestEval if the new depth is higher.
// Fires the vindication callback if the engine's best move matches the game move.
onEngineUpdate(best: string, eval_: { cp?: number; mate?: number; depth: number }): void;

// Register a callback fired once when the game move is vindicated.
// Cleared when advancing to the next candidate.
onVindication(cb: () => void): void;
```

Internal state to add to the retro session object:
- `_liveBestMove: string | undefined` — starts as `cand.bestMove`
- `_liveBestDepth: number` — starts as the batch analysis depth (0 if unknown)
- `_liveBestEval: { cp?: number; mate?: number; depth: number } | undefined`
- `_vindicationCb: (() => void) | undefined`
- `_vindicated: boolean` — prevents firing the callback more than once per candidate

`onEngineUpdate()` logic:
1. If `eval_.depth <= _liveBestDepth`, ignore (older or equal depth).
2. Update `_liveBestMove`, `_liveBestDepth`, `_liveBestEval`.
3. If `best === cand.gameMove` (UCI) and `!_vindicated`, set `_vindicated = true`,
   call `_vindicationCb?.()`.

On `jumpToNext()` / `reset()`: clear `_vindicated`, `_vindicationCb`, reset live state
to the new candidate's batch values.

### What NOT to change

- Do not change candidate selection, scoring, or session lifecycle beyond the above.
- Do not change `retroMoveHandler.ts` — that is CCP-633.
- Do not change `retroView.ts` — that is CCP-634.
