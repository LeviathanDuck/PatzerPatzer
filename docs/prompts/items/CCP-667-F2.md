# CCP-667-F2 — Fix Prompt Editor Flicker Before Entering Edit View

## Summary

Follow up on `CCP-667-F1`.

Right now, when the user clicks `Edit` in prompt detail, the UI briefly shows a strange
intermediate preview/state before it settles into what looks like the edit viewer.

On further inspection, the problem is worse than a visual flicker:
- the user may see an intermediate preview-like state with another apparent edit step
- even after pressing through that extra step, the resulting surface may still not behave like a
  real editable text editor
- the user should not have to click through a preview phase just to reach actual editable text

This should feel immediate and intentional:
- clicking `Edit` should transition directly into the edit viewer
- the detail panel should not briefly flash a partial preview or odd intermediate state

## Diagnosis Target

Inspect the current prompt detail edit transition in the generated dashboard implementation.

The likely issue is that the detail view is being fully re-rendered in a way that briefly
shows a non-editing state before the edit-state shell is mounted.

Confirm the exact cause in the current code before changing behavior.

## Required Fix

Implement the smallest safe fix so prompt detail edit mode enters cleanly.

Expected behavior:
- click `Edit`
- prompt detail goes straight into a real editable stylized text viewer
- no strange transient preview / flicker / intermediate read-only redraw appears first
- once edit mode is entered, the prompt body must actually accept text editing immediately

Constraints:
- keep this scoped to prompt detail edit-mode entry behavior
- do not redesign the whole editor model again
- do not regress locked metadata behavior from `CCP-667`
- do not regress the line-by-line stylized editor behavior from `CCP-667-F1`
- do not change sprint panel editing unless the same shared helper must be adjusted safely
