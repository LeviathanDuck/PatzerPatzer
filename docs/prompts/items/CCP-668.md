# CCP-668 — Add Skipped Prompt Lifecycle And Dashboard Control

## Summary

Add a first-class `skipped` prompt lifecycle state so queued prompts can be intentionally
removed from the runnable queue without being mistaken for active work later.

This must be bulletproof:
- skipped prompts must never appear as runnable queue candidates
- skipped prompts must never be startable/completable/reviewable
- if a user later asks an agent what prompts are available in queue, skipped prompts must
  never be included

## Scope

Implement the smallest complete workflow change needed to support skipping a prompt safely:

1. Add a first-class skipped prompt state in the prompt registry workflow.
2. Add a skip command and dashboard action.
3. Make skipped prompts non-runnable everywhere.
4. Remove skipped prompts from all queue / next-up / runnable surfaces.
5. Update all canonical prompt workflow docs that need to know this behavior exists.

## Required Behavior

### Registry / lifecycle model

- introduce `status: skipped`
- skipped prompts must also use `queueState: not-queued`
- add `skippedAt`
- optionally add `skipReason` if needed for clarity

Hard rules:
- skipped prompts are still real tracked prompt records
- skipped prompts are not superseded prompts
- skipped prompts are not runnable prompts

### Command behavior

Add a lifecycle command:

```sh
npm run prompt:skip -- <PROMPT_ID>
```

Expected behavior:
- confirms the prompt exists
- rejects reserved / superseded / already-reviewed / already-skipped prompts
- transitions the prompt to:
  - `status: skipped`
  - `queueState: not-queued`
  - `skippedAt: <now>`
- regenerates prompt workflow artifacts via the normal refresh path

### Dashboard behavior

In prompt detail view:
- add a `Skip This Prompt` button in the top-right action area
- this button should only appear for prompts that are still skippable
- clicking it must require an explicit confirmation step before mutating state

After skip:
- prompt must no longer show as queued
- prompt must no longer appear in Next Up / urgent queue / queue list surfaces
- prompt detail should clearly show that it was skipped and when

### Queue safety

Skipped prompts must never appear in:
- `CLAUDE_PROMPT_QUEUE.md`
- dashboard queued views
- next-up helpers
- urgent queue helpers
- any queue-derived “available prompt” surfaces

### Lifecycle safety

After a prompt is skipped, the following must fail cleanly:
- `npm run prompt:start -- <PROMPT_ID>`
- `npm run prompt:complete -- <PROMPT_ID>`
- `npm run prompt:review -- <PROMPT_ID> ...`
