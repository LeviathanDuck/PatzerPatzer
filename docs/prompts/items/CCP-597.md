# Wire eval diff into RetroCandidate + silent background eval for best-move positions

## Context

Part of CCP-595 (eval diff for retro move quality). This is child prompt 2/3.
Depends on CCP-596 (`src/analyse/evalDiff.ts`) being complete.

The eval diff utility exists. Now we need to:
1. Add `evalDiff` to `RetroCandidate` so the data travels with each exercise.
2. Populate it at session-build time from existing cache data.
3. For candidates where the played-move eval is missing or too shallow, silently
   request a background engine evaluation — invisible to the user, no arrows,
   no board updates, no ceval display.

## Files to touch (2)

- `src/analyse/retro.ts` — add `evalDiff` field to `RetroCandidate`; populate
  it inside `buildRetroCandidates` using `computeEvalDiff` from CCP-596.
- `src/analyse/retroMoveHandler.ts` — after the session resolves a candidate
  (win or fail), trigger the silent background eval pass for the next
  candidate if its diff is not yet computed.

## Task 1 — Extend `RetroCandidate`

Add to the `RetroCandidate` interface in `src/analyse/retro.ts`:

```typescript
/**
 * Eval difference between the engine best move and the played move.
 * Populated at session-build time when both parentEval and nodeEval are
 * in the cache. May be absent for candidates built before analysis completed
 * or where the played position was never evaluated.
 */
evalDiff?: EvalDiff;   // import from '../analyse/evalDiff'
```

In `buildRetroCandidates`, after creating each candidate, attempt to compute
the diff:

```typescript
const diff = computeEvalDiff({
  parentEval: parentEval,
  nodeEval:   nodeEval,
  playerColor: candidate.playerColor,
});
if (diff) candidate.evalDiff = diff;
```

`parentEval` and `nodeEval` are already in scope from the existing candidate
logic. `nodeEval.loss` is the preferred source (see CCP-596); no new lookups
are needed for the common case.

## Task 2 — Silent background eval for missing/shallow diffs

Some candidates will have `evalDiff === undefined` at session start — either
because the played position is not in the cache (game not yet fully analyzed),
or because the cached eval is too shallow to be reliable.

When these candidates are encountered (either at session start or when advancing
to the next candidate), a one-shot silent engine evaluation of the played-move
position should be requested.

### What "silent" means

- The engine runs on the played-move FEN at a fixed depth (use `retroConfig.missedMateDistance * 6`
  or a hardcoded depth of 18 — whichever is greater, capped at 20).
- The result updates `evalCache` for the played position's path.
- No arrows are drawn, no ceval panel updates, no `redraw()` is triggered.
- The eval is stored by calling `evalCache.set(candidate.path, result)` and then
  `candidate.evalDiff = computeEvalDiff(...)`.
- The engine must be restored to analysis mode (or its previous state) after the
  one-shot completes.

### Triggering the background eval

Add an exported function:

```typescript
/**
 * Request a silent background evaluation of the played-move position for
 * a retro candidate. Used when `candidate.evalDiff` is absent or the
 * source eval is too shallow.
 *
 * The function is a no-op if:
 *   - the engine is currently in analysis or play mode (do not interrupt)
 *   - evalDiff is already populated with confidence === 'cp'
 *   - a background eval for this path is already in flight
 *
 * @param candidate - the candidate whose played position needs evaluating
 * @param getEval   - evalCache getter
 * @param setEval   - evalCache setter (called when result arrives)
 * @param onDone    - called after diff is computed; caller responsible for redraw
 */
export function requestRetroBackgroundEval(
  candidate: RetroCandidate,
  getEval: (path: string) => PositionEval | undefined,
  setEval: (path: string, ev: PositionEval) => void,
  onDone: (diff: EvalDiff) => void,
): void
```

This function lives in `src/analyse/retro.ts` (it is analysis-domain logic,
not board/engine wiring). It should:
- Check that `engineMode === 'analysis'` and engine is idle before proceeding
- Use `enterPlayMode` / `exitPlayMode` is NOT appropriate here — this is
  analysis-mode eval, not play mode. Use the existing UCI path directly
  via `protocol.setPosition(candidate.fenBefore)` + `protocol.goDepth(depth)`,
  treating the result as an analysis eval.
- Actually: since this is a one-off analysis eval, prefer using `evalCurrentPosition`
  seam if it can be directed to a specific FEN, or simply call the existing
  batch eval infrastructure with a single-item queue.
- If the engine is not idle, skip silently (the candidate will display without
  a diff until the next session rebuild).

### Where it is called from

In `src/analyse/retroMoveHandler.ts`, in the after-move hook, after the
current candidate's outcome is resolved (feedback === 'win' or has transitioned
to 'fail'), call `requestRetroBackgroundEval` on the NEXT candidate (not the
current one — the current candidate's diff should already be computed from cache).
This way the eval is ready before the user gets to the next exercise.

Also call it once when the retro session is first built (for the first candidate),
if its diff is absent.

## What this does NOT do

- Does not change the analysis board's eval display
- Does not show any engine output or arrows for the background eval
- Does not interrupt live analysis if the engine is running
- Does not introduce a second engine instance

## Validation

- `npm run build` passes
- `npx tsc --noEmit` passes
- Starting a retro session with a reviewed game: first candidate has `evalDiff`
  populated (check via `console.log` — diff should be present for positions in cache)
- Engine does not show extra arrows or ceval output during a retro session
