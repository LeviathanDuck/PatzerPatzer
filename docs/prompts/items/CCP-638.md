# CCP-638 — Review button splits into queue-position icons when queue is active

Read and follow:
- `/Users/leftcoast/Development/PatzerPatzer/CLAUDE.md`
- `/Users/leftcoast/Development/PatzerPatzer/AGENTS.md`

## Context

When no review queue is active, each game row in the games list shows a single Review
button. When a review queue IS active, that button should split into two icon-buttons
that occupy the exact same total width — one to add the game to the top of the queue
("Review Next") and one to add it to the bottom ("Add to Queue").

**Important prerequisite:** it is not confirmed that a review queue with ordered
add-to-top / add-to-bottom semantics currently exists in the codebase. The executor
must audit the current queue implementation first. If add-to-top / add-to-bottom
queue positioning does not exist, that infrastructure must be engineered as part of
this prompt before the UI can be wired up.

## Behaviour spec

**No queue active (current behaviour — do not change):**
- Single Review button per game row
- Clicking it starts a review for that game immediately

**Queue active — single game row:**
- The Review button is replaced by two icon-buttons side by side
- Combined width = exactly the width of the single Review button
- Each icon-button takes 50% of that width
- Left icon: add to top of queue ("Review Next")
- Right icon: add to bottom of queue ("Add to Queue")
- Each icon has a visible hover tooltip:
  - Left: "Review next"
  - Right: "Add to queue"
- Clicking either button adds that game to the correct queue position and gives
  immediate visual feedback on the row (e.g. brief highlight or icon state change)

**Queue active — multiple games selected:**
- The existing multi-select behaviour in the games list must be preserved
- When one or more games are selected and the user clicks either icon on any selected
  row, ALL currently selected games are added to the top or bottom of the queue
  respectively, in the order they appear in the list
- This mirrors standard multi-select queue behaviour in media player apps

## Visual design direction

Style these buttons to follow the visual language already established in the app.
The primary external reference for the interaction pattern and icon style is the
**Pocket Casts** podcast app's "Play Next" / "Play Last" queue buttons, which appear
in their track list rows. Key characteristics to mirror:

- Icon-only buttons — no label text inside the button
- Icons are compact and vertically centered within the row height
- "Add to top" icon: conveys "move to front" — a common approach is an arrow-with-line
  pointing up (⬆ with a bar at top) or a similar queue-top metaphor. Choose whichever
  reads most clearly at small size from standard Unicode or an SVG icon.
- "Add to bottom" icon: same metaphor pointing down
- Buttons have a subtle hover state (background tint or opacity change) consistent with
  other interactive elements in the games list
- No border between the two buttons, or a very thin divider — they should read as a
  unified control occupying the original button footprint
- Inactive / disabled state should be visually distinct if a game is already in the
  queue at either end (optional — implement only if trivially available from queue state)

Do not introduce new design patterns. Match existing button colors, border-radius,
and spacing conventions already present in the app's game list rows.

## Pre-implementation checklist

Before writing any code:

1. **Audit the review queue** — does a review queue with ordered positioning currently
   exist? Check `src/engine/batch.ts` and any queue module. Specifically confirm:
   - Is there a concept of an ordered queue of games awaiting review?
   - Is there an API to add a game to the top vs bottom of that queue?
   - If not, document what exists and what needs to be added.

2. **Read the current games list view** — find where the Review button is rendered in
   `src/games/view.ts`. Understand the button's current width, class, and click handler.

3. **Read multi-select handling** — find where selected games are tracked. Understand
   what state or set holds the current selection so the bulk-add path can use it.

4. **Confirm queue-active detection** — find the predicate or flag that indicates a
   review queue is currently running. This gates the button split.

5. **Check Lichess reference** — look for any analogous split-button, queue-action, or
   icon-button pattern in `ui/analyse/src/` or `ui/lib/src/` that informs the
   implementation. Lichess study/playlist patterns may be relevant.

## Implementation scope

**If the queue infrastructure does not exist:**
Engineer the minimum required queue primitives (ordered list, add-to-top, add-to-bottom)
before touching the UI. Keep this isolated to the appropriate module — do not expand
`main.ts`. Document what was added.

**UI changes (always required):**
- `src/games/view.ts` — conditional rendering of single button vs split icon pair;
  multi-select bulk-add path
- `src/styles/main.scss` — styles for the split icon buttons

Do not use raw DOM manipulation. All rendering uses `h()` Snabbdom vnodes.

Width constraint: wrap both buttons in a `div` constrained to the same width as the
original Review button; each child `button` at `width: 50%`.

Hover tooltip: use the `title` attribute on each `button` element. This is the standard
approach used elsewhere in the app for icon-button affordances.

## What NOT to do

- Do not change multi-select selection logic — only the action triggered by the icon buttons
- Do not redesign the game row layout beyond the button cell
- Do not add labels or text inside the icon buttons

## Validation

- With no queue active: single Review button is unchanged
- With a queue active:
  - Review button is replaced by two equal-width icon buttons
  - Combined width matches the original Review button width
  - Hovering each button shows the correct tooltip text
  - Clicking the left icon on a single-selected row adds that game to the top of the queue
  - Clicking the right icon on a single-selected row adds that game to the bottom
  - With multiple games selected, clicking either icon adds all selected games in list order
    to the top or bottom respectively
- No layout shift or wrapping at normal game list widths
