## Context

The sprint card grid in `docs/prompts/dashboard.html` (generated by
`scripts/generate-prompt-dashboard.mjs`) currently shows:

- Title + status badge header
- Three metric bars: Plan / Execution / Implementation
- Segmented bar showing task state distribution
- Completion summary text
- Pills: task count, linked count, unreviewed count, audits, next steps, next phase, normalization warning, dependencies

Several issues have been identified:

1. The segmented bar and the three metric bars are redundant — both derived from the same task state data.
2. "Plan" is a misleading label — it actually means % of tasks with a linked prompt (prompt coverage).
3. The pills row is cluttered — `X tasks` and `X linked` add little on top of the bars.
4. `needs normalization` is buried at low contrast in the pills — it is a warning state and deserves visual prominence.
5. There is no age/recency signal on the card.
6. No left-border status colour for fast Kanban-style scanning.
7. Unreviewed prompt count looks the same as neutral pills even when it signals action needed.

## Goal

Polish the sprint card grid in `scripts/generate-prompt-dashboard.mjs` to be
more scannable and informative without adding clutter.

## Changes required

### 1. Remove the segmented bar

Drop the `<div class="progress-bar">` + `progressSegments` call from each card.
Keep only the three metric bars. The segmented bar is redundant with the metrics.

### 2. Rename "Plan" label to "Prompts"

Change the label on the first metric bar from `Plan` to `Prompts` to accurately
reflect that it measures prompt-coverage, not planning completeness.

### 3. Left border status colour

Add a coloured left border to each `.sprint-card` that maps to the sprint status.
Use the same colour palette as the existing `.sprint-status--*` badges:
- Needs Prompts / Incomplete Start State → amber (`#7a6940`)
- Ready to Start → purple (`#7b62b6`)
- In Progress → blue (`#416b82`)
- Completed Needs Full Review / Completed: With Issues → orange-brown (`#7e6536`)
- Completed: Reviews Passed / Completed: Review Passed / Completed → green (`#3c6a4b`)

Implement as inline `border-left: 3px solid <colour>` on the card element,
or via a CSS class `sprint-card--status-<slug>` with matching rules.

### 4. Warning card tint when normalization required

When `sprint.showNormalizationWarning` is true, add a class (e.g.
`sprint-card--warn`) to the card that applies a subtle amber background tint
and amber border colour, overriding the status border. This makes the card
visually distinct from a healthy card at a glance.

### 5. Unreviewed count as warning badge

When `sprint.promptsUnreviewedCount > 0`, render it as an amber/orange
warning pill (new class e.g. `sprint-pill--warn`) instead of the default
neutral pill. Apply a distinct colour so it reads as "action needed."

### 6. Trim the neutral pills

Remove `X tasks` and `X linked` from the pills row — this information is
already conveyed by the progress bars. Keep:
- Unreviewed count (as warning badge when > 0)
- Audit count (if any)
- Next steps count (if any)
- Next phase name (if any)
- Dependency pills

### 7. Relative date footer

Add a small "Updated X days ago" (or "today" / "yesterday") line at the
bottom of each card using `sprint.updatedAt`. Keep it at the lowest visual
weight (e.g. `font-size: 0.7rem; color: var(--text-dim); opacity: 0.7`).

Use a client-side function — do not recompute in the generator.

## Hard constraints

- Do not change sprint registry logic or computed fields
- Do not change the sprint detail overlay
- Do not change the prompt tab or prompt cards
- Scope all changes to the sprint card CSS and the `renderSprints` function
