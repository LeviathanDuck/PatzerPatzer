# CCP-650 — Design sprint normalization warning and mixed task presentation

Read and follow:
- `/Users/leftcoast/Development/PatzerPatzer/CLAUDE.md`
- `/Users/leftcoast/Development/PatzerPatzer/AGENTS.md`

## Context

Codex has already added backend support for two important sprint-detail states:

1. a normalization warning when a sprint is not reliable enough for next-phase prompt generation
2. mixed next-phase task coverage where some tasks already have prompts and some still need them

Those backend states are present, but the UI treatment is basic. The user wants Claude to help
design this so the warning state and mixed-task state are understandable instead of feeling like
raw diagnostic output.

## Relevant file

- `scripts/generate-prompt-dashboard.mjs`

Generated output to inspect:
- `docs/prompts/dashboard.html`

## Goal

Improve the UI treatment for:
- the sprint normalization warning
- mixed covered/uncovered task presentation inside sprint detail

## Hard constraints

- Do not alter sprint semantics
- Do not change the rules for when the normalization warning appears
- Do not change the rules for when the next-prompt panel appears
- Keep this prompt scoped to warning/mixed-state presentation only

## Backend contract already implemented

Consume the existing backend fields as-is:
- `normalizedStructure`
- `showNormalizationWarning`
- `showNextPromptPanel`
- `nextPhaseTasks`
- `tasksNeedingPrompts`
- `nextPhaseCoveredTasks`

Do not introduce new workflow rules here.

## Validation

Run:
```sh
npm run dashboard:generate
```

Open `docs/prompts/dashboard.html` and verify:
- non-normalized sprint detail clearly communicates that an audit/normalization step is required
- mixed covered/uncovered task situations are easier to understand
- warning state feels meaningful without overpowering the rest of the view
