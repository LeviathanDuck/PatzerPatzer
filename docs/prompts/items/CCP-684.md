# CCP-684 — Add Per-Phase Prompt Generation Copy Buttons To Sprint Detail

## Summary

Add a large copy action to each phase shown in sprint detail view:

- button text: `Generate Next Prompts Template`
- behavior: copy the prompt-generation instructions for that exact phase

This must not be implemented as a cosmetic copy button that reuses the current sprint-level
next-phase template for every phase.

Right now the sprint system reliably generates a template only for the sprint's current
`nextAvailablePhase`. That is not sufficient for this feature because the user wants a button on
every listed phase, and the copied template must stay accurate for the exact phase the button
belongs to.

## Goal

Make every phase in sprint detail view capable of copying a phase-specific prompt-generation
template that is accurate for that phase's current task/linkage state.

## Required Behavior

For every phase shown in sprint detail view:
- show a prominent copy button labeled `Generate Next Prompts Template`
- clicking the button copies a phase-specific template for that exact phase
- the template must be derived from current sprint registry truth, not hardcoded prose
- the template must include:
  - sprint id
  - sprint title
  - source sprint doc
  - exact phase id
  - exact phase title
  - tasks in that phase still needing prompts
  - tasks in that phase already covered by linked prompts
  - linked prompt ids for already-covered tasks
  - rules telling the agent not to recreate already-covered prompts unless needed
  - rules telling the agent to use exact sprint linkage fields when creating prompts

## Accuracy Rules

The copied template must be accurate for the selected phase, not only the next available phase.

Use sprint registry data as source of truth for:
- phase/task structure
- task current state
- linked prompt ids

If a phase has no tasks needing prompts:
- still allow the phase button to exist
- but the copied template must say that no tasks in that phase currently need prompts
- and it must still list already-covered tasks for audit/reference context

If a phase is not yet ready because an earlier phase is incomplete:
- the template should still reflect the exact phase state truthfully
- but it should make that dependency clear instead of pretending the phase is currently ready

## Implementation Notes

This likely requires backend generation changes, not just dashboard rendering changes.

Relevant code paths likely include:
- `/Users/leftcoast/Development/PatzerPatzer/scripts/sprint-registry-lib.mjs`
- `/Users/leftcoast/Development/PatzerPatzer/scripts/generate-prompt-dashboard.mjs`
- `/Users/leftcoast/Development/PatzerPatzer/docs/prompts/dashboard.html`

Current limitation to address:
- `buildNextPromptsTemplate(...)` currently works only from `sprint.nextAvailablePhase`
- sprint detail currently exposes only the sprint-level next-phase action panel, not per-phase templates

Preferred approach:
- add a reusable phase-specific template builder in sprint registry/dashboard data
- attach a generated template to each phase record in sprint detail data
- render the per-phase copy button from that phase-specific data

## Validation

- open a sprint detail view and confirm every listed phase shows a `Generate Next Prompts Template` copy button
- click the button for the sprint's current next available phase and confirm the copied template matches that phase's current task/linkage state
- click the button for a later not-ready phase and confirm the copied template references that exact phase and accurately reflects its current readiness/task state
- click the button for a completed or already-covered phase and confirm the copied template does not invent missing prompt work
- confirm the existing sprint-level prompt-generation panels still render correctly and do not regress
