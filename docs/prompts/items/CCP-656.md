# CCP-656 — Opponent Import: Skip Importing Screen, Show Progress in Tree Build Bar Slot

## Context

Currently when a user clicks "Import Games" in the opponent research details form, the view
switches to a full-page `importing` step (transfer animation + game count + Cancel button)
that replaces the entire body area. After the fetch and tree build complete, a separate `done`
step is shown before the user can reach the board view.

The desired flow:
1. User fills in details (username, source, filters) — **unchanged**
2. User clicks "Import Games" → view **immediately** switches to the opening-tree board view
3. During game fetch: a progress bar appears in the **same slot** where the tree build bar
   lives (`renderTreeBuildBar` position, inside the session panel above the engine section)
4. During tree build: existing `renderTreeBuildBar` takes over seamlessly
5. Import complete → user is already in the board view, no `done` step needed

Research basis: CCP-654 confirmed this is feasible in 2 files with a new `_isFetching` flag.

## Task

### 1. Add fetch-phase state to `src/openings/ctrl.ts`

Add two new pieces of state alongside the existing `_importProgress`:
- `_isFetching: boolean` — true from the moment "Import Games" is clicked until the fetch
  completes (or errors/cancels); false at all other times
- No new progress counter needed — `importProgress()` already tracks game count during fetch

Export getters:
- `isFetching(): boolean`

Update `executeResearchImport` (in `src/openings/import.ts`) to:
- Set `_isFetching = true` right before the fetch starts (after validation)
- Set `_isFetching = false` when the fetch completes, errors, or is cancelled
- Also update `cancelImport()` and `resetImport()` to clear `_isFetching = false`

### 2. Switch to board view on import start in `src/openings/view.ts`

In `executeResearchImport` (called from the Import button click handler), after the existing
validation passes and the fetch is about to begin, call `setActiveTool('opening-tree')` so
the view immediately transitions to the board area.

The simplest hook: since `executeResearchImport` lives in `src/openings/import.ts` and already
calls `setImportStep('importing')`, change `setImportStep('importing')` to instead call
`setActiveTool('opening-tree')` and `setImportStep('idle')` — the import step is no longer
needed to track the fetch phase once the board view owns it.

Alternatively, the click handler in `renderDetailsStep` can call `setActiveTool('opening-tree')`
before delegating to `executeResearchImport`. Either approach is fine; pick whichever is cleaner
after reading the code.

### 3. Render fetch progress bar in `renderOpeningTreeTool` (`src/openings/view.ts`)

At line 1792 (currently `treeBuilding() ? renderTreeBuildBar() : null`), extend the condition:

```
isFetching() ? renderFetchBar() : treeBuilding() ? renderTreeBuildBar() : null
```

Add `renderFetchBar()` as a small private function that reuses the same vnode structure as
`renderTreeBuildBar` but shows fetch-phase content:
- An indeterminate bar (full width, pulsing CSS, or just 100% fill) since total game count
  is not known upfront
- Label: `"Fetching games… N found"` using `importProgress()` for the count (0 = "Connecting…")
- The bar should visually match `renderTreeBuildBar` so the transition is seamless

### 4. Handle the no-collection state in `renderOpeningTreeTool`

During the fetch, `activeCollection()` is null (no collection exists yet). The board and
session panel must still render. Check that `renderOpeningTreeTool` does not crash or blank
when called with `collection = null` — it already accepts `collection: ResearchCollection | null`
so this should be safe, but verify and add any needed null guards.

### 5. Remove `importing` and `done` steps

In `renderImportWorkflow` (`src/openings/view.ts`):
- Remove `step === 'importing' ? renderImportingStep(redraw) : null`
- Remove `step === 'done' ? renderDoneStep(redraw) : null`
- Remove `renderImportingStep` and `renderDoneStep` function bodies

In `executeResearchImport` (`src/openings/import.ts`):
- Remove `setImportStep('importing')` and `setImportStep('done')` calls
- On success: call `resetImport()` (which already sets step back to `idle`) and
  `setActiveTool('opening-tree')` if not already there
- On error: set `_isFetching = false`, keep step at `'details'` so the form is shown with
  the error message (existing `setImportError` behavior)

### 6. Cancel during fetch

`cancelImport()` already aborts the in-flight request and sets step back to `'details'` with
an error message. Add `_isFetching = false` to that path. The Cancel affordance moves: instead
of the full-page `importing` step Cancel button, add a small Cancel link/button in the
`renderFetchBar` vnode (next to the label) that calls `cancelImport(); redraw()`.

## Scope

- `src/openings/ctrl.ts` — add `_isFetching`, `isFetching()`, update `cancelImport()` /
  `resetImport()`
- `src/openings/import.ts` — set/clear `_isFetching`, remove `setImportStep('importing')`
  and `setImportStep('done')`, add `setActiveTool` on success
- `src/openings/view.ts` — add `renderFetchBar()`, update tree-build-bar slot condition,
  remove `renderImportingStep` / `renderDoneStep`, add Cancel to fetch bar

3 files maximum.
