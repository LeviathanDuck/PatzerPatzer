# CCP-667 — Lock Prompt Metadata Lines In Dashboard Editor

## Summary

Fix the prompt detail dashboard editor so registry-owned metadata lines are truly locked in the UI,
not just rejected at save time.

The current bug is:
- prompt detail edit mode visually shows the metadata block separately
- but the editable textarea still contains those lines
- so users can edit `Prompt ID`, `Task ID`, `Parent Prompt ID`, `Source Document`,
  `Source Step`, and `Execution Target`
- save is then blocked by backend validation, but the UI contract is still wrong

The expected behavior is:
- locked metadata remains visible inline at the top of the editor
- those lines are not editable
- the editable surface starts only after the locked metadata block
- diff preview and save behavior continue to work correctly for the editable body

## Scope

Inspect and update the prompt-detail editor path in at least:
- `/Users/leftcoast/Development/PatzerPatzer/scripts/generate-prompt-dashboard.mjs`
- `/Users/leftcoast/Development/PatzerPatzer/server.mjs`
- any prompt workflow docs that now misdescribe the actual editor behavior

## Requirements

1. Keep registry-owned metadata visibly present in edit mode.
2. Make those lines non-editable in the UI, not merely save-blocked later.
3. Preserve the current save flow:
   - same active `CCP-###`
   - superseded archive handling
   - `lastEditedAt`
   - full artifact regeneration
4. Preserve the current diff preview semantics for the editable portion of the body.
5. Preserve the localhost-only guidance in static `file://` mode.
6. Do not regress prompt copying, cancel/reset behavior, or archived-history display.

## Validation

- open an editable prompt in dashboard detail view and confirm metadata lines are visible but cannot be modified
- edit the first editable line after metadata and confirm live diff preview updates correctly
- save a valid body edit and confirm save still succeeds end to end
- attempt to change only editable body content and confirm backend metadata-mismatch error is not triggered
- confirm static `file://` mode still shows localhost instructions instead of pretending edit/save works
