# CCP-672 — Replace Line Editor With Textarea Plus Diff-On-Confirm

Prompt ID: CCP-672
Task ID: CCP-672
Source Document: (audit conversation, 2026-04-02)

## Context

The dashboard's inline prompt/sprint editor uses a custom 4-state line-by-line editor (~350 LOC client-side)
with context/removed/replacement/added line tracking, pairKey pairing, and per-line `<input>` elements.
This is overengineered for editing prompt bodies. It should be replaced with a standard `<textarea>`.

CCP-671 must be completed first (fixes the escaping bug that prevents the current editor from working at all).

## Task

In `scripts/generate-prompt-dashboard.mjs`, replace the line editor with a textarea-based editor.
Both prompt editing and sprint panel editing must use the same approach.

### Remove (from the generated JS)

- `createLineEditorState()` — 4-state line array creation
- `cleanLineEditorBody()` — filter/join from line array
- `activateLineEditorLine()` — paired removal/replacement logic
- `deleteContextLine()` — context→removed transition
- `updateLineEditorText()` — per-line text mutation
- `insertLineEditorLineAfter()` — add line after
- `removeEditableLine()` — remove added/replacement
- `restoreRemovedLine()` — restore with pairKey handling
- `renderLineEditor()` — per-line HTML grid rendering
- `nextLineEditorId()` / `lineEditorUid` — ID generation
- All line-level event handlers: `activate`, `delete-context`, `restore`, `remove-editable`, line `input` events, Enter-key line splitting
- All `.line-editor-*` CSS classes

### Replace with

**Edit state** (for both prompts and sprint panels):
```
{
  original: string,   // body text when editing started
  current: string,    // current textarea contents
  confirming: boolean // false=editing, true=showing diff confirmation
}
```

**Editing view:**
- Locked metadata rendered as read-only styled text above the textarea (same split logic as today)
- A `<textarea>` containing the editable body portion
- Buttons: "Revert to Original" (resets textarea to `original`), "Save" (enters confirmation), "Cancel" (exits edit mode, warns if changed)

**Confirmation view (when `confirming: true`):**
- Diff display using the existing `buildPromptDiffOps()` LCS algorithm (already fixed by CCP-671)
- Color coding: green for added lines, red for removed lines, neutral for context
- Buttons: "Confirm Save" (sends to server), "Back to Edit" (returns to textarea), "Decline / Revert" (reverts to original and exits edit mode)

**Behavior:**
- "Revert to Original" is available at any time during editing — resets textarea to `original`
- "Decline" in the confirmation step reverts to original and exits edit mode entirely
- The `buildPromptDiffOps` function stays (it's a clean ~30-line LCS, not part of the line editor)
- The `renderPromptDiffHtml` and `summarizePromptDiff` functions stay (they render the diff output)

### Files to touch

1. `scripts/generate-prompt-dashboard.mjs` — editor rewrite
2. `docs/prompts/dashboard.html` — regenerated output

## Validation

1. Open an editable prompt in the dashboard (served via `node server.mjs`)
2. Click Edit — locked header is read-only, body appears in a textarea
3. Make changes in the textarea
4. Click "Revert to Original" — textarea resets to the original body
5. Make changes again, click "Save" — diff confirmation appears with green/red lines
6. Click "Back to Edit" — returns to textarea with changes preserved
7. Click "Save" again, then "Decline / Revert" — reverts to original, exits edit mode
8. Click "Save" again, then "Confirm Save" — server saves, prompt refreshes
9. Repeat steps 2-8 for a sprint panel editor — behavior should be identical
10. Verify no `.line-editor-*` CSS or line state functions remain in the generated HTML
