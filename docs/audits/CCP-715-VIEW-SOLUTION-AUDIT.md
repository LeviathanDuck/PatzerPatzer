# CCP-715 — Audit: LFYM View-Solution Reveal Flow

_Produced: 2026-04-03_

## 1. Solution Path Strategy

### Current behavior (broken)

`src/analyse/retroView.ts` lines 228-231:
```typescript
retro.viewSolution();
if (cand) navigate(cand.path);
```

`cand.path` is the path to the **mistake node** (the wrong move the user played in the game).
This navigates to the mistake, not the solution. The panel shows the correct solution text
(`bestSan`) but the board stays on the wrong position.

### Available tree operations

`src/tree/ops.ts` provides:
- `nodeAtPath(root, path)` — lookup a node by path
- `childById(node, id)` — find a child by 2-char ID
- `addNode(root, path, node)` — add a child node at a path (no-op if child with same ID exists)

Node IDs are generated by `scalachessCharPair(move)` from `chessops/compat` (used in
`src/tree/pgn.ts:64` and `src/board/index.ts:266`).

### Recommended approach: use `playUciMove()`

`src/board/index.ts` exports `playUciMove(uci: string)` (line 282). This function:
1. Parses the UCI string into a chessops Move
2. Checks if a child node with that move already exists (by matching `c.uci`)
3. If yes: navigates to the existing child
4. If no: creates a new TreeNode with the correct `scalachessCharPair` ID, FEN, SAN, etc.,
   adds it to the tree via `addNode()`, and navigates to it

**This is the exact pattern needed.** The view-solution handler should:
1. First navigate to `cand.parentPath` (the puzzle-start position)
2. Then call `playUciMove(cand.bestMove)` to play the solution move on the board

This handles both cases (existing variation vs new node) and reuses the established tree
mutation path. It also triggers `navigate()` internally, which calls `evalCurrentPosition()`,
`syncArrow()`, and all the standard post-navigation hooks.

**Alternative considered and rejected:** Manually computing the solution node ID via
`scalachessCharPair` and constructing a path. This duplicates logic already in `playUciMove`
and requires parsing the UCI, creating a Chess position, calling `makeSanAndPlay`, etc. —
all of which `playUciMove` already does.

### Path sequence for the handler

```typescript
// 1. Set feedback state
retro.viewSolution();

// 2. Navigate to puzzle start (position before the mistake)
navigate(cand.parentPath);

// 3. Play the solution move (creates node if needed, navigates to it)
playUciMove(cand.bestMove);

// 4. Reveal engine + eval (see section 2)
```

**Important:** `navigate()` has a same-path guard (line 559: `if (path === ctrl.path) return`).
If `ctrl.path` is already at `cand.parentPath`, step 2 is a no-op. But `playUciMove` reads
`ctrl.node` (the current node), so `ctrl.path` MUST be at `cand.parentPath` before calling it.
The current handler navigates to `cand.path` (wrong), so `ctrl.path` won't be at `parentPath`
— step 2 will execute correctly.

Edge case: if the user has already navigated to `parentPath` (e.g., via the move list) before
clicking "View the solution", the `navigate()` call is a no-op, but `ctrl.node` is already
correct for `playUciMove`. Safe.

## 2. Engine Visibility Call Sequence

After navigating to the solution position, the engine should be visible and analyzing.

### Current dual-flag system

Two independent flags gate engine output during LFYM:

| Flag | Location | Controls |
|---|---|---|
| `retroVisibleEngineEnabled` | `src/ceval/view.ts:65` | Eval bar, pearl, PV box rendering |
| `_guidanceRevealed` | `src/analyse/retroCtrl.ts:298` | Arrow shapes in `buildArrowShapes()` |

Both must be `true` for full engine output to appear. This is the split-control problem
documented in the known issue.

### Required call sequence for reveal

```typescript
// a. Set retroVisibleEngineEnabled = true
//    No direct setter exported; must set via the module state.
//    Either: export a setter function, or use the existing toggle path.
//    Simplest: export a `setRetroVisibleEngineEnabled(v: boolean)` from ceval/view.ts

// b. Set _guidanceRevealed = true
retro.revealGuidance();  // already exists (retroCtrl.ts:409)

// c. Sync arrows (so retroHidden recalculates to false)
syncArrow();

// d. Engine is already running (forced on at LFYM session start).
//    navigate() already called evalCurrentPosition() in step 3 above.
//    No additional eval call needed.
```

**However:** `playUciMove()` internally calls `navigate()` which calls `evalCurrentPosition()`,
`syncArrow()`, and `redraw()`. So steps (c) and (d) are already handled by step 3 in section 1,
**provided** the flags are set BEFORE `playUciMove` is called. The rendering gates in `main.ts`
(lines 922-929) read the flags at render time, which happens during `redraw()` inside
`navigate()`.

### Recommended sequence (final)

```typescript
retro.viewSolution();                    // feedback = 'view', mark solved
retro.revealGuidance();                  // _guidanceRevealed = true
// Set retroVisibleEngineEnabled = true  // (needs new setter or direct set)
navigate(cand.parentPath);               // go to puzzle start
playUciMove(cand.bestMove);              // play solution, navigate to result
// navigate() inside playUciMove triggers: syncArrow, evalCurrentPosition, redraw
```

### Setter needed

`src/ceval/view.ts` currently exports:
- `isRetroVisibleEngineEnabled()` — getter
- `resetRetroVisibleEngineUi()` — resets to false

It does NOT export a setter to set it to `true`. **A new export is needed:**
```typescript
export function setRetroVisibleEngineEnabled(v: boolean): void {
  retroVisibleEngineEnabled = v;
}
```

Or combine with the existing reset into a general setter.

## 3. Analysis Unlock

### Current state

`onJump()` in `src/analyse/retroCtrl.ts` (lines 416-438) handles navigation during LFYM:

```typescript
if (_feedback === 'win' || _feedback === 'view') {
  return; // ← early return, no offTrack detection
}
```

**When `feedback === 'view'`, navigation is already unlocked.** The `onJump()` early return
means the user can freely navigate the move tree without being flagged as "off track."

### No additional state change needed

The `viewSolution()` method (line 380) sets `_feedback = 'view'`, which is sufficient to
unlock navigation. No additional flag or method is required.

### Board interaction

Board moves are handled by `src/board/index.ts` which calls `applyMoveToTree()` → `navigate()`.
The `onJump()` call inside `navigate()` (line 567) will hit the `'view'` early return. The user
can make moves, explore variations, etc. This matches the target behavior.

## 5. Proposed Seam Split: CCP-715-F1 and CCP-715-F2

### CCP-715-F1 — Fix: LFYM View-Solution Board Navigation

**Scope:** Fix the "View the solution" click handler to navigate to the solution position.

**Files:** `src/analyse/retroView.ts` only (1 file)

**Changes:**
- In the `renderSkipOrView()` function (line 228-231), change the click handler from:
  ```typescript
  retro.viewSolution();
  if (cand) navigate(cand.path);
  ```
  to:
  ```typescript
  retro.viewSolution();
  if (cand) {
    navigate(cand.parentPath);
    playUciMove(cand.bestMove);
  }
  ```
- Add `playUciMove` to the imports from `src/board/index.ts`

**Does NOT include:** Engine reveal, Show Engine button removal, or ceval view changes.

**Scope:** After view-solution navigates to the correct position (CCP-715-F1), reveal engine
output and remove the redundant "Show Engine" button.

**Files:** `src/analyse/retroView.ts`, `src/ceval/view.ts`, `src/main.ts` (3 files)

**Changes:**

   - Add `setRetroVisibleEngineEnabled(v: boolean)` export

2. In `src/analyse/retroView.ts`:
   - In the view-solution click handler (modified by F1), add:
     ```typescript

     setRetroVisibleEngineEnabled(true);
     ```

     before the `navigate()` call
   - Remove the "Show Engine" button (lines 597-604) and `...showEngineBtn` insertion
   - Remove `onRevealGuidance` from the function signature

### Rationale for this split

- F1 is a pure navigation fix (1 file, minimal risk)
- F2 is the engine visibility unification (3 files, moderate risk)
- F1 can be tested and verified independently before F2
- If F2 causes issues, F1 still provides the correct board navigation
- Both together deliver the full fix described in the known issue
